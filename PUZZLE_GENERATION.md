# Puzzle Generation Pipeline

## Overview

KnotQuite's puzzles are now generated by a three-stage pipeline:

1. **Authoring** (Claude API or LM Studio local LLM) — Creates candidate puzzles
2. **Validation** (mechanical checks) — Rejects bad categories, obscure words, structural errors
3. **Curation** (manual spot-check) — Human review of sample before shipping

This replaces the old approach of randomly combining scraped Wikipedia categories, which produced unsolvable puzzles like "Fijian expatriate rugby union players in Italy" with names like WAME LEWARAVU.

## 🎮 Generating 3 Months of Puzzles (New!)

To generate a complete 3-month supply of puzzles with daily and random pools completely separated:

### Quick Start

```bash
# Set up your API key (Claude API via Anthropic)
export ANTHROPIC_API_KEY=sk-your-key-here

# Generate 450 puzzles (90 daily + 360 random)
npm run generate:puzzles:3months

# Validate the puzzles
npm run validate:puzzles

# Build the daily rotation
npm run build:daily-order

# Test in the app
npm run dev
```

### What Gets Generated

- **90 Daily Puzzles** (IDs 1-90): One per day for 3 months
  - Used exclusively by the Daily mode
  - Appears in `dailyOrder.json` for epoch-based rotation
  - Cycled in order, never repeats within the 90-day period
  
- **360 Random Puzzles** (IDs 91-450): Four per day for random selection
  - Used exclusively by Random mode
  - Excluded from daily rotation
  - **Completely separate from daily puzzles**

### Puzzle Separation

Each puzzle in `puzzles.json` has a `type` field:
```json
{
  "id": 1,
  "theme": "...",
  "groups": [...],
  "redHerring": {...},
  "type": "daily"  // or "random"
}
```

The app enforces this separation:
- **Daily mode** only picks from puzzles with `type: 'daily'` (via `dailyOrder.json`)
- **Random mode** only picks from puzzles with `type: 'random'` (filters out daily puzzles)
- A puzzle will **never** appear in both modes

### Why Separation?

Users should never see the same puzzle in both their daily rotation and random selection. This separation guarantees:
- Daily players get a stable, predictable experience
- Random players always have fresh puzzle discovery
- No surprises when switching between modes

## Quick Start

### 1. Install LM Studio

Download LM Studio from [lmstudio.ai](https://lmstudio.ai).

### 2. Load a Model

- Open LM Studio
- Search for and download a model (e.g., **Gemma 3** — pick the largest your machine can run comfortably, since this runs offline on your computer only)
- Load the model in the Developer tab
- Verify the server is running at `http://localhost:1234` (shown in LM Studio's Developer tab)

### 3. Generate Puzzles

```bash
npm run generate:puzzles 100
```

This generates up to 100 validated puzzle candidates, saving them to `src/data/puzzles.candidates.json`.

Monitor the output for:
- `✓ Valid: puzzle <id>` — accepted candidate
- `✗ Invalid: puzzle <id>` — rejected (shows reason)

### 4. Validate the Full Bank

```bash
npm run validate:puzzles
```

Shows:
- Count of valid/invalid puzzles in `src/data/puzzles.json`
- First 50 failures with reasons
- High-recurrence words (reused too often)

### 5. Review Candidate Puzzles

After generation, spot-check a random sample (~30) from `src/data/puzzles.candidates.json` against this checklist:

- [ ] Theme genuinely unifies all 4 groups (not just concatenated category names)
- [ ] Category names read like Connections categories, not Wikipedia lists
- [ ] No obscure proper nouns (names unknown to typical adults)
- [ ] Red herring is a plausible false category, not a structural coincidence
- [ ] Difficulty progression makes sense (yellow easiest, purple trickiest)
- [ ] No duplicate words within or across groups
- [ ] All words are common English (no jargon, no ultra-niche terms)

If the sample fails noticeably more than the validator reports, tighten the prompt or validator thresholds before generating more.

### 6. Commit to Production

Once validated and spot-checked:

```bash
cp src/data/puzzles.candidates.json src/data/puzzles.json
npm run build:renderer  # verify app still builds
npm run dev            # test in Electron — play a Daily and Random puzzle
```

Then commit and push.

## Troubleshooting

### "LM Studio server not reachable at http://localhost:1234"

- Is LM Studio running? (Check the app window is open)
- Is a model loaded? (Check the Developer tab — it should show a running server)
- Try restarting LM Studio
- Verify LM Studio is listening: `curl http://localhost:1234/v1/models` in a terminal

### "No valid JSON found in response"

The model generated text that didn't parse as valid JSON. This happens with smaller models. Try:
- Using a larger model (e.g. Gemma 3B or 7B instead of 2B)
- Lowering `temperature` in `lmstudio.config.json` (more deterministic, less creative)
- Rerunning the generator (retry logic is built in)

### Validator reports "999/1000 invalid" on the current puzzles.json

This is expected — the current data is the low-quality auto-generated set. The validator is working correctly. Use it to validate the new pipeline's output, not the old data.

## Ongoing Workflow

### After First 3-Month Generation

Once you've generated the initial 450 puzzles:

1. **Validate**: `npm run validate:puzzles` — check for structural errors
2. **Spot-check**: Manually review a sample (~30 puzzles) from `src/data/puzzles.json`
3. **Build daily order**: `npm run build:daily-order` — creates deterministic daily rotation
4. **Test**: `npm run dev` — play both Daily and Random modes
5. **Commit**: `git add . && git commit -m "Add 3-month puzzle supply"`

### Incrementally Adding More Puzzles

If you want to extend beyond 3 months later:

**Option A: Generate individual puzzles (easier, incremental)**
```bash
npm run generate:puzzles 50  # Uses Claude API to add 50 more
npm run validate:puzzles
npm run build:daily-order
```

**Option B: Regenerate the full 3-month supply (nuclear option)**
```bash
rm src/data/puzzles.json  # Start fresh
npm run generate:puzzles:3months
```

### Important: Pool Separation Rules

**Never mix daily and random puzzles!**

- If you add new puzzles via `npm run generate:puzzles`, manually assign IDs and `type` fields:
  - IDs 1-90 with `type: 'daily'` → daily rotation only
  - IDs 91+ with `type: 'random'` → random selection only
  
- If you use `generate:puzzles:3months`, IDs are auto-assigned correctly

The app enforces separation at runtime:
- `getDailyPuzzle()` uses only IDs in `dailyOrder.json`
- `pickRandomPuzzle()` filters by `type: 'random'`

## Advanced: Customizing the Authoring Prompt

Edit `scripts/lib/authoring-prompt.md` to:
- Add topic constraints ("Generate puzzles about food, music, and sports")
- Adjust difficulty guidelines
- Add/remove examples

Re-run `npm run generate:puzzles` — it reads the latest prompt each time.

## Advanced: Adjusting Validation

To make validation stricter or looser, edit `scripts/validate_puzzles.js`:

- **Loosen obscurity check**: Reduce the "3/4 proper nouns" threshold
- **Tighten obscurity check**: Lower the common-words cutoff or extend the allowlist
- **Add new checks**: Insert validation logic before the final pass/fail decision

Then re-run `npm run validate:puzzles` to see the impact.
